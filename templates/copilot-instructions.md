## Code Quality Guidelines

### Style
- Follow the existing code style consistently
- Use meaningful and descriptive names for variables, functions, and classes
- Keep functions focused on a single responsibility
- Maintain consistent indentation and formatting

### Documentation
- Add clear comments for complex algorithms or business logic
- Write comprehensive docstrings for public APIs
- Keep inline comments up to date with code changes
- Document any non-obvious design decisions

### Testing
- Write unit tests for new functionality
- Ensure edge cases are covered
- Keep tests readable and maintainable
- Run tests before committing

### Best Practices
- Handle errors gracefully
- Validate inputs appropriately
- Consider security implications
- Write defensive code
- Avoid premature optimization
- Keep code DRY (Don't Repeat Yourself)

## Git Practices
- Write clear, descriptive commit messages
- Keep commits atomic and logical
- Review your own changes before committing
- Ensure code works before pushing

## Review Checklist
Before suggesting code:
- [ ] Does it follow the project's existing patterns?
- [ ] Is it readable and maintainable?
- [ ] Are there tests?
- [ ] Is it secure?
- [ ] Is error handling appropriate?
- [ ] Is documentation needed?
